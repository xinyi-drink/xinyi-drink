# 意图路由

本文件承接 `SKILL.md` 的意图判断细节，功能不变，只用于让主规则更短、更清晰。

## 路由表

| 用户表达 | 判断 | 动作 |
| --- | --- | --- |
| “大礼包怎么领取”“怎么参与活动”“怎么领福利” | 活动流程说明 | 未提供手机号时直接请用户发送微信小程序【新一咖啡】绑定手机号；不要先要求用户去登录/注册，也不要解释 `no_reward_config` |
| “帮我领取新一Skill福利”“我想领福利”“参加活动” | 活动领取 | 已提供手机号时调用 `claim_reward.py`；未提供手机号时先请求【新一咖啡】绑定手机号 |
| “有什么活动”“有哪些福利” | 活动总览 | 调用 `recommend_drink.py --use-saved-mobile` 获取聚合上下文；只说明 Skill用户大礼包，不扩展到其它商品信息 |
| 询问参与活动所用手机号、已保存手机号、历史参与手机号、领取礼包手机号、绑定活动手机号或当前活动手机号等语义，不限于示例原文，如“新一咖啡参与活动手机号是什么”“之前参与活动的手机号”“我上次领礼包用的手机号”；或问“本机保存的手机号是什么”“新一咖啡缓存手机号是什么”“活动状态是什么” | 本地手机号状态查询 | 调用 `recommend_drink.py --show-mobile-status`；只读本地状态文件，不请求后端，不查 Agent 记忆、历史记忆或对话记忆，不回答成固定的活动手机号 |
| “这个手机号领过了吗”“我是不是参加过”“换个手机号”“手机号改成…”“重新绑定手机号”“用另一个手机号” | 活动状态/换号预检 | 用户本轮提供手机号时先调用 `recommend_drink.py --show-mobile-status --candidate-mobile <手机号>`；未提供手机号时调用 `recommend_drink.py --show-mobile-status`；只读本地状态文件，不请求后端。状态查询和换号预检不能直接调用 `claim_reward.py`；只有用户明确确认继续领取或同步后，才调用 `claim_reward.py --mobile <手机号>` |
| “我登录小程序了”“确认领取”“继续领取”“用这个手机号领取” | 活动领取/同步 | 用户明确要领取或同步活动结果时调用 `claim_reward.py --mobile <手机号>`；未提供手机号且已有缓存时调用 `claim_reward.py --use-saved-mobile`；由脚本同步本地活动状态并输出标准化结果，不要绕过脚本直接调用 `/skill/xinyi/claim` |
| 个人订单、下单、购买、消费、喝过、买过、点过、取餐、制作中或历史记录，例如“我买过多少杯”“我都定了哪些饮料”“我喝了多少咖啡”“我的饮品做好了吗”“看我的历史订单”“我买过什么” | 订单查询 | 调用 `query_orders.py --use-saved-mobile` 查询用户订单；用户本轮提供手机号时改用 `--mobile`；问正在进行中订单加 `--status 2`，问历史/已完成订单加 `--status 4`，否则不传 status 查全部；不要求用户说出“订单”两个字；订单信息只能基于本次查询结果，不能预估、估算或模糊处理 |
| “这个 skill 是什么”“你能做什么”“怎么用 xinyi-drink” | Skill 介绍 | 直接说明四个固定主用途：领取Skill用户大礼包、查询及分析个人历史订单、查询菜单及饮品热量、查询门店及等候时长；未参加活动时可补留资文案 |
| “手机号换成…”“手机号改成…”“重新绑定手机号”“重新输入手机号”“用另一个手机号”“刚才不是这个手机号” | 更换手机号 | 先调用 `recommend_drink.py --show-mobile-status --candidate-mobile <手机号>` 做本地只读预检；若本机缓存已确认参与，直接说明不能更换手机号重复领取。若本地未确认已参与，必须先让用户明确确认继续领取或同步，再调用 `claim_reward.py --mobile <手机号>` |
| “新一咖啡有哪些门店”“望京店目前有多少杯待做，等待时间多久” | 门店查询 | 调用 `fetch_stores.py`；当前不获取定位，回答时只展示本次查询到的全部门店，不按距离排序 |
| “某某饮品热量多少”“有哪些不太甜的果茶”“有哪些清爽不太甜的茶饮” | 商品/饮品信息 | 调用 `recommend_drink.py --query <问题>` 获取聚合上下文 |

## 触发优先级

1. 明确提到新一品牌或新一相关内容时，优先使用本 Skill。
2. 没提新一但上下文已经在新一饮品、门店或活动场景中，也可以使用本 Skill。
3. 泛饮品、泛咖啡、泛门店问题只有在上下文足够明确时才使用。
4. 其他品牌、竞品、通用营养、配方知识和纯闲聊不使用本 Skill。

## 手机号触发边界

- 参与活动、查询活动状态、活动总览、订单摘要和口味偏好分析时，可以请求或复用手机号。
- 用户只是在问参与活动所用手机号、已保存手机号、历史参与手机号、领取礼包手机号、绑定活动手机号、当前活动手机号或活动状态时，使用 `--show-mobile-status` 本地读取，不请求后端，不查 Agent 记忆、历史记忆或对话记忆。
- 普通推荐、门店查询和普通商品查询不要主动索要手机号，也不要复用缓存手机号。
- 用户明确要求更换手机号、改手机号、重新绑定、重新输入手机号或使用另一个手机号时，也必须通过 `claim_reward.py --mobile <手机号>` 同步状态；只有当前缓存手机号未确认领取成功时，才允许换号。
